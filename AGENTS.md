# AGENTS.md — Project Brutality 2022 (main mod)

This file orients coding agents to the Project Brutality add-on that lives in this folder. Read it before editing anything here.

**Recent work (player-facing + maintainer):** see **`CHANGELOG.md`** in this folder. It uses [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) sections (`[Unreleased]`, Added / Changed / Fixed / Removed / Documentation, plus optional pending follow-ups). **After you ship a notable fix or feature**, append a bullet under the right heading in `[Unreleased]` (or under a new dated heading when you tag a release). **`git log`** remains the full history; the changelog is the curated summary for humans and agents.

## 1. What this is

- **Project Brutality 2022** — a large gameplay and content overhaul for *Doom* / *Doom II*, built for **GZDoom** (engine line currently tracked by [UZDoom](https://github.com/UZDoom/UZDoom/tree/4.14.3)).
- Loose folder layout, **not** PK3-packed. GZDoom loads the directory directly (drag the folder onto `gzdoom.exe`, or pass it with `-file`).
- ZScript version pin: `version "4.5.0"` at the top of `ZSCRIPT.zc` — minimum GZDoom ~4.5; development generally targets the latest stable GZDoom.
- Custom player class: **`PB_Doomer`** (locked by `NoRandomPlayerClass = True` in `gameinfo.txt`).
- The **Glory Kills** and **Monster Pack** content was previously shipped as sibling add-on folders and has been folded into this mod. There is nothing external to load on top of the main folder anymore — treat the Glory Kill / pinata system and the extra MP monsters (Crackodemon, Hellduke, Hellsmith, Director, Horrorspawn, Hunger, Aracnorb Queen, Helemental, Hierophant, ESoul, ZombieTank/Plasma/Missile/Elite, PB_ZombieFlyer, etc.) as first-class main-mod content.

## 4. Architecture notes that catch agents off guard

- **Mixed DECORATE + ZScript.** Most actors are still DECORATE (`actors/**/*.dec`). Newer systems — event handlers, HUD UI, wheel UI, gore, PlayerPawn movement, execution GUI — are ZScript (`zscript/**/*.zc|.zsc|.txt`). **Do not port DECORATE to ZScript wholesale.** Follow the convention of the file you're editing.
- **Inventory tokens as flags.** A large block at the top of `DECORATE` defines tiny `Inventory` actors (`EvadeCheck`, `ExecutionToken`, `IsDead`, `PlayerWheelOpen`, `CantFire`, `NoZombieGrenade`, `BrutalCounter`, `FallingHeight`, `TargetIsACyberdemon`, etc.) that are used as booleans/counters through `A_GiveInventory` / `CountInv` / `CheckInventory`. Scan these before adding a new one — many already exist. **Barrel grab / carry** tokens for the Staging-style set (incl. flaming barrels) live in **`actors/Decorations/Barrels/Core/NukageBarrelTokens.dec`** — use **`HasBurningBarrel` / `GrabbedBurningBarrel`** (not `*Flame*`) for compatibility with **`PLAYER.dec`** / **`BaseWeapon`**.
- **Event handlers are registered via MAPINFO**, not in ZScript. See the `GameInfo { AddEventHandlers = ... }` block in `zmapinfo.txt`. Examples: `pb_ExecutionHandler`, `PB_EventHandler`, `MBlurHandler`, `PB_WheelBSSBlurHandler`, `NashGoreHandler`, `BDv22GoreHandler`, `BDv22GoreMergeHandler`, `BPv10GoreHandler`, `PB_DeathGoreBoostHandler`, `TiltPlusPlusHandler`, `DeathFadeBootstrap`, `DEDashJumpHandler`, `GrapplingHookHandler`, `SpeedoMeterHandler`, `WallSlideHandler`, `PB_SpecialWheelHandler`, `HatExtravaganza`, `PB_HitFeedback_Handler`, `HelmetDropHandler`, `DamNumEventHandler`, **`PB_TacticalFeelHandler`**, `PB_KillstreakMessageHandler`, `PB_KillstreakHandler`, `PB_KillstreakRareFlare`, `PB_InstantWeaponSwitchHandler`, `CSSGEvent`, **`PB_FrozenSolidHandler`** — **always prefer `zmapinfo.txt` as source of truth**; the list grows with features.
  Adding a handler requires both a ZScript class **and** a new `AddEventHandlers` line.
- **Editor numbers** come from the reserved **23000–23133** range in the `DoomEdNums` block of `zmapinfo.txt`. Add a new entry there when introducing a placeable actor; do not reuse numbers.
- **ACS workflow.** Edit `.acs` in `SRC/`, recompile to `.o` in `ACS/` with **ACC** (matching the GZDoom version), and make sure the lump name is listed in `LOADACS.txt`. Never hand-edit `.o` files. Run **`acc.exe`** (e.g. from SLADE’s bundled ACC) with **`-i`** pointing at the folder that contains **`zcommon.acs`**. Common outputs: `SRC/GloryHUD.acs` → `ACS/GloryHUD.o`, `SRC/GloryChain.acs` → `ACS/GloryMelee.o` (lump name differs from source filename — see `LOADACS.txt`).
- **Player class is fixed** to `PB_Doomer`. It is defined in `actors/Player/PLAYER.dec` and extended by `PlayerPawnBase : PlayerPawn` in `zscript/PlayerPawn.zc`. `gameinfo.txt` enforces `NoRandomPlayerClass = True`.
- **Custom weapon flags** live in `EPBWeaponFlags` (inside `ZSCRIPT.zc`): `PBWEAP_KEEPYOFFSET`, `PBWEAP_ISPISTOLSILENCERSTATE`, `PBWEAP_UNLOADED`. Weapons extend `PB_Weapon` (DECORATE) on top of `PB_WeaponBase` (ZScript).
- **ZScript-only weapons: inherit `PB_Weapon`, not only `PB_WeaponBase`.** In `actors/Weapons/BaseWeapon.dec`, `Actor PB_Weapon : PB_WeaponBase` owns **`SelectFirstPersonLegs`** (registers `KickHandler_Overlay`, `Melee_Equipment_Handler_Overlay`, `Equipment_Toggle_Handler_Overlay`, first-person legs, `DoKick`, etc.) and then **`A_Jump(255, "SelectContinue")`**. A ZScript `class MyGun : PB_WeaponBase` **does not** run that chain unless you wire it. **Recipe:** (1) `class MyGun : PB_Weapon`. (2) In **`Select`**, after per-weapon setup / barrel clears, **`goto SelectFirstPersonLegs`** (do **not** `goto` straight into a local `SelectPre` / ad-hoc bring-up). (3) Expose a **`SelectContinue:`** state with your fatality check, `PB_WeapTokenSwitch`, `PB_RespectIfNeeded`, and **`goto SelectAnimation`** (or equivalent). If an older file used **`SelectPre`**, rename it to **`SelectContinue`**. (4) **Do not** paste a second copy of **`SelectFirstPersonLegs`** in the same class — it drifts from `BaseWeapon.dec`; drop it and use the parent’s state. (5) **`FlashPunching` / flash-kick states** must return to a ready state (**`goto Ready3`** is typical) — never end with **`stop`**, or the weapon can get stuck and anything that needs **`player.ReadyWeapon == self` + a steady ready loop** (e.g. **`DoEffect()`** for **alt-hold shields / barrier PSprites** on `PB_NeoHMG`, `PB_ArgentSith`, `PB_BeamKatana`, etc.) will fail. *Shipped ZScript examples that follow this pattern:* `BDPBattleRifle`, `PB_CSSG`, `PB_Excavator`, `PB_NeoHMG`, `PB_ArgentSith`, `PB_BeamKatana`.
- **Punch-flash psprite layer (`FlashPunching` dispatch).** `QuickMelee` in both `actors/Weapons/BaseWeapon.dec` (`GoMeleeInstead`) and `zscript/Weapons/BaseWeapon.zc` now dispatches **`FlashPunching`** on **`PSP_FLASH`** (was a generic layer 4 in older PB 2022) to match PB Staging. Entry does `A_ClearOverlays(PSP_FLASH, PSP_FLASH, false)` *before* `A_Overlay(PSP_FLASH, "FlashPunching")`, and the `QuickMelee` exit into `Ready3` does the same clear. The net effect: **`FlashPunching`** may end with **either `Stop` or `Goto Ready3`** without the old duplicate-sprite bug — use `Goto Ready3` for symmetry with **`FlashKicking`** (which already returns to ready). Overrides that launch `FlashPunching` themselves (e.g. `BeamKatana_Weapon.zs`, `ArgentSith_Weapon.zs` `QuickMelee`) must use `PSP_FLASH` too and perform the same exit-time clear before transitioning. Loops like `RiotShield`'s `ReadyShield` are **not** safe targets of a `Goto` from `FlashPunching` because the ready state would loop on `PSP_FLASH` instead of `PSP_WEAPON`; those keep `Stop`. Full writeup of the engine behaviour, why layer 4 produced the duplicate, and how PB Staging's `PSP_FLASH` dispatch sidesteps it, is in `docs/punch_flash_psp_flash.md`.
- **`A_ClearOverlays` has a hidden `safety=true` default that silently SKIPS `PSP_WEAPON` and `PSP_FLASH`.** Native signature is `A_ClearOverlays(int sstart = 0, int sstop = 0, bool safety = true)`. With `safety=true` (the default, which DECORATE and ZScript both inherit), the engine explicitly `continue`s past `id == PSP_STRIFEHANDS || id == PSP_WEAPON || id == PSP_FLASH` — so `A_ClearOverlays(PSP_FLASH, PSP_FLASH)` is a silent no-op, not a clear. To actually destroy PSP_WEAPON/PSP_FLASH via this function you **must** pass `false` as the third argument, e.g. `A_ClearOverlays(PSP_FLASH, PSP_FLASH, false)`. There is no runtime warning and no DECORATE lint for this. Every PSP_FLASH clear in the `FlashPunching` plumbing passes `safety=false`. Regular layer-range clears like `A_ClearOverlays(10, 11)` are unaffected because those layers are not in the safety allow-list. Engine source: UZDoom `src/playsim/p_pspr.cpp` `A_ClearOverlays`. Alternative PSP_FLASH destroyers: `A_GunFlash("Null")`, `player.SetPsprite(PSP_FLASH, null)`, `A_Overlay(PSP_FLASH, "Null")`.
- **Monster base classes.** DECORATE monsters extend `PB_Monster` (or its DECORATE wrappers) and usually declare `Replaces <VanillaActor>`. ZScript monster logic lives in the abstract `PB_Monster` class in `zscript/Monsters/PBMonster.zc` (hitbox handler, `A_SmartChase`, execution handler).
- **Global constants.** Defined in `ZSCRIPT.zc`: `MAXITERATIONS`, `STAT_NashGore_Gore`, `STAT_PB_BULLETS`, `C_TID`, `MAX_R`, `ADJUST_R`, `VIEW_HEIGHT`.
- **Cat's Frozen Addon fold — frozen-solid statues + frost monsters.** **`PB_FrozenSolidHandler`** (always on; no menu toggle) listens on **`WorldThingDied`** and, when the fatal damage type is **`Ice` / `Freeze` / `Frost` / `IceBeam`**, replaces select **`PB_*`** monsters (Imp, Baron, Mancubus family, Cacodemon, Pinky demon, Revenant, Cyberdemon family, Mastermind family — see `zscript/Monsters/CatsFrozen/PB_FrozenSolidHandler.zc`) with **`PB_FrozenSolid*`** ZScript actors that use 8-angle sprites under **`SPRITES/Monsters/CatsFrozenFrozen/`**. The handler **`Destroy()`**s the dying monster actor after spawning the statue to avoid piling on top of the engine ice-corpse path; it does **not** rip out PB's core **`+ICEDEATH`** / IceVile plumbing. Frost-themed combat monsters (**`PB_FrostBaron`**, **`PB_Cryocubus`**, **`PB_FrostBrain`**, ZScript **`PB_Cryotron`**) live under **`actors/Monsters/CatsFrozen/`** + **`zscript/Monsters/CatsFrozen/PB_Cryotron.zc`** — **no `replaces`**, rolled by pack spawners with **`pb_disablenewenemies`** plus optional **`pb_NoFrostBaron`** / **`pb_NoCryocubus`** / **`pb_NoFrostBrain`** / **`pb_NoCryotron`** (**`MonsterPackSettings`** / Simpler Monster Settings bundles). Asset notes: **`SPRITES/Monsters/CatsFrozenFrozen/ASSET_NOTES.txt`**, **`SPRITES/Monsters/CatsFrozen/`**, **`SPRITES/Effects/CatsFrozenIce/`**.
- **Cat's Frozen Addon — equipment wheel (Slice 4).** Eight inventory-backed modes (**`WW_FreezenadeSelected`**, **`WW_PB_CF_*`**) extend **`specialWheel_Equipment`** / **`WheelSliceIsUsable`** / **`BaseWeapon.dec`** (`SwitchEquipment`, **`UseEquipment`**, **`WheelCancelEquipment`**) alongside Molotov / grenades. Uses Slice 1 cryo actors (**`PBCF_CryocubusSlowdown`**, **`PBCF_CryoWind`**, **`PBCF_IceOrbCryo`**, snow clouds, etc.) without duplicating classes. Deployables inherit **`PBCF_FriendlyDeployableBase`** for player-sourced damage filtering (same idea as **`FamiliarBase`**). **No** `pb_No<Equipment>` cvars and **no** new **`MENUDEF`** rows for these pieces (standing PB directive); optional backpack loot only.
- **Cat's Frozen Addon — weapons (Slice 5).** Three **`PB_Weapon`** DECORATE guns: **`PB_CryoElectroRifle`** (slot **8**, **`PB_CryoCells`** + **`PB_Cell`** reload; cryo **`PBCF_IceOrbCryo`** vs electric **`LightningMissile2`** from **`Stormcast.dec`**; **`PB_CryoElectroRifle_ElectricMode`** + **`specialWheel_PB_CryoElectroRifle`** / **`Select_PB_CryoElectro_*`** same pattern as **`HellPistoler`**), **`PB_CryoCannon`** (slot **9**, **`PB_CryoCannonCells`**, **`PBCF_CryoWind`**), **`PB_CryoASG`** (slot **3**, **`PB_Shell`**, **`PBCF_CryoPellet`**). **No** `pb_No<Weapon>` cvars / **no** **`MENUDEF`** spawn toggles for this fold (explicit directive); spawner branches are unconditional alongside existing weapon pools.
- **PDA toggle + Blocky tile-grid layout (`pb_pda_blocky`).** **Toggle:** the same **`netevent openpda`** opens AND closes the PDA. **`PB_EventHandler.NetworkProcess`** in **`zscript/PBPDA.zc`** branches on `PDAMode` (open path freezes / sets `PROP_TOTALLYFROZEN` / plays `PDAOPEN`; close path mirrors `exitpda` cleanup and plays `PDABACK`). A `transient int LastPDAToggleTic` field plus a `gametic - LastPDAToggleTic < 6` debounce gate prevents an open->close->open race when the bind fires twice within a few engine tics. **`KEYCONF.txt`** bind unchanged (`defaultbind BACKSLASH openpda`). **Blocky layout:** server bool **`pb_pda_blocky`** (default **on**) gates a Gearbox-inspired (`gb_BlockyView`, `zscript/gearbox/display/blocky_view.zs`) paged tile renderer for the Weapons / Monsters codex sections. New helper section in **`zscript/PBPDA.zc`** with banner comment `=== Blocky-format PDA layout helpers ===`: `BlockyEnabled` / `GetBlockyMetrics` / `DrawBlockyTile` / `DrawBlockyLabel` / `BlockySectionTotal` / `ApplyBlockyLayout`, plus **`ResolveBlockyTileIcon`** / **`DrawBlockyTileIcon`**: weapons resolve **`Inventory.Icon`** → **`Weapon.AltHUDIcon`** (class default, same idea as `DrawPB2022WeaponPickupOverlay`) → **`ClassNameICON`** lump → first non-**TNT1** frame from **`Spawn`** (`BlockyFirstFrameOfState`, 48-state cap); monsters use the first **`PDAMONSTT`** class with **`Spawn`** → **`See`** → **`Idle`** (`FindState('Idle')`) walks in that order. Resolve results are cached via parallel `transient ui Array<String>` + `Array<TextureID>` pairs (**`blockyWeaponIconKeys`** / **`blockyWeaponIconVals`** + monster sibling) — `Map<>` is unavailable on PB's ZScript `version "4.5.0"` pin (Maps require >= 4.10). Linear `Find()` lookup; n is bounded by catalogue size so O(n) is fine. Positive hits revalidate via `cached.IsValid()` and the matching `[idx]` is `Delete()`d from both parallel arrays + re-resolved if the underlying texture went stale (e.g. `flushtex` mid-session). Confirmed-iconless classes are tracked in two sibling **`transient ui Array<String>`** "missed" arrays (**`blockyWeaponIconMissed`** / **`blockyMonsterIconMissed`**, presence-as-sentinel) so the negative-cache fast-path doesn't impersonate a stale positive hit and doesn't force a state-walk every frame. All six arrays are **`Clear()`**'d when **`InitializeUiObjects`** runs after **`uiObjects.Size() < 1`** (one-shot per save load). Weapons that ship a curated portrait (e.g. **`PB_DarkMatterRifle`** → **`PLCUA0`**) declare it via DECORATE **`Inventory.Icon`** so the highest-priority slot in the resolve chain hits before any state-walk fallback. Unknown portraits draw a muted **`BigFont`** **`?`** (`Font.CR_DARKGRAY`). **Cast syntax constraint:** every `class<X>` cast in this resolve path uses the parenthesized C-style form **`(class<X>)(nm)`** — unparenthesized `class<X>(nm)` is rejected by UZDoom 4.14.3 ZScript with `Unexpected 'class'` (canonical PB pattern: `(class<Ammo>)(type)` in `BaseWeapon.zc::GetParentAmmo`, `(class<Actor>)(...)` in `MetalSniper.zs`). Hovered or selected tiles apply a small sine **`gametic`** scale pulse to the icon draw only (`PB_PDA_BLOCKY_ICON_PULSE_AMP` × `sin(gametic * 0.15)`), not **`ImagePos`/`ImageSize`**. Tiles drawn with **`Screen.Dim`** + **`Screen.DrawLine`** rectangles (Gearbox's `gb_box` / `gb_weap` / `gb_cor` lumps live inside Gearbox's own assets, not as loose patches in this tree). Geometry: **`PB_PDA_BLOCKY_COLS = 5`**, **`PB_PDA_BLOCKY_ROWS = 8`**, padding scaled by screen height; tiles fill the left half of the screen so the description pane (anchored at `Screen.GetWidth() / 2`) keeps its existing right-side render. Hidden-until-found rows (`dontdraw == true`) stay hidden in blocky mode too (locked tiles are not shown to avoid spoiling discovery). **Pagination:** existing PDA Up/Down arrows (`objecttype` 4 / 5) are repurposed as page-up / page-down in blocky mode (legacy "nudge desc-pane Y while held" preserved when blocky is off); page indices `pdaBlockyWeapPage` / `pdaBlockyMonsterPage` are `transient ui int`. **`ApplyBlockyLayout`** rewrites every `objecttype 3` (weapon) / `objecttype 6` (monster) UiObject's `ImagePos` / `ImageSize` to its tile slot; off-page tiles get pushed to `(-9999, -9999)` so the existing UiTick mouse hit-test (MouseX/Y vs ImagePos+ImageSize) skips them automatically without needing a separate `pageHidden` flag. The function is called once at the end of **`InitializeUiObjects`** (rebuilds positions on first open / save load) and again at the top of every **`RenderOverlay`** frame in PDA mode (resolution-change / page-advance reflow). Off = legacy stacked-text list (preserved as fallback). ## 5. Common-change recipes

**Add a new monster (DECORATE path)**
1. Create `actors/Monsters/<Tier>/<Name>.dec` extending `PB_Monster` (or an appropriate sibling).
2. Add `#include "actors/Monsters/<Tier>/<Name>.dec"` near the other monsters in `DECORATE`.
3. Reserve a `DoomEdNum` in `zmapinfo.txt` (next free in 23000–23133 range).
4. Register sounds in `SNDINFO.txt` (e.g. under a `// --- monsters ---` style section or with a new header comment).
5. Add sprites under `SPRITES/`, brightmaps in `BMAP/` + bindings in `doommonsters.bm`, and any dynamic lights in `GLDEFS`.
6. If the monster should spawn in vanilla levels, either use `Replaces` on the actor or hook into the appropriate spawner under `actors/SPAWNERS/MonsterSpawners/`.

**Add a new weapon**
1. Create `actors/Weapons/Slot<N>/<Name>.dec` extending `PB_Weapon` (from `actors/Weapons/BaseWeapon.dec`).
2. `#include` it under the matching section in `DECORATE`. If the weapon references a PBWP ammo alias (`PB_Shell`, `PB_HighCalMag`, `PB_Cell`, `PB_CellPack`) by name in state-action args, place the `#include` **after** the `PBWP_CompatAmmo.dec` block near line ~157, not in the trailing weapon block — see the **"DECORATE include order"** bullet in §4.
3. Reserve a `DoomEdNum` in `zmapinfo.txt`.
4. Add ammo (if needed) in `actors/Items/Ammo/` and pickup strings to `language.enu`.
5. Register sounds in `SNDINFO.txt` in the `PBWeapons` (or new weapon) section.
6. Add sprites/graphics and any `TEXTURES.*` patches.
7. If the weapon is ZScript-only (`class …` in `zscript/Weapons/`), make it **`class MyGun : PB_Weapon`** and wire **`Select` → `SelectFirstPersonLegs` → `SelectContinue`** (see the **ZScript-only weapons** bullet in §4). For shared helpers only, `PB_WeaponBase` in `zscript/Weapons/BaseWeapon.zc` still applies.
8. If the weapon can spawn from a map spawner, add a `server int pb_No<Weapon>Weapon` (or matching existing `pb_No*` naming) in `CVARINFO`, mirror it in **both** `WeaponSpawns` and `AddOnToggles` in `MENUDEF.txt` with `SpawnOnOff`, and gate every spawn path with `GetCVar` / ACS toggles the way existing weapons do.
9. If the weapon uses one or more ammo types, add an explicit `IsSelected <ClassName>` block in `SBARINFO.txt` (do not rely on the generic fallback for shipped weapons).

**Fold a weapon add-on into this mod (agent checklist)**
1. **Add-on layout:** Prefer an add-on folder that already contains top-level **`Sounds/`** and **`Sprites/`** (or equivalent) so all audio and graphics can be copied without hunting loose lumps. If upstream only ships a PK3, unpack to that shape before porting. Example fold: **XM-21 Sniper** — `actors/Weapons/Slot4/XM21.dec`, `sounds/combat/weapons/XM21/`, `SPRITES/WEAPONS/Slot4/XM21/`, the `SNDINFO.txt` (PBWeapons) block, `pb_NoPB_XM21Weapon` + chaingun spawner `SpawnPB_XM21`, `IsSelected PB_XM21` in `SBARINFO.txt`, `PB_PICKUP_PB_XM21` / `PB_WPNHINT_PB_XM21` in `language.enu`. Newer add-on **`PB_Projectile` / `PB_HighCalMag`** types were mapped to `FastProjectile` + **`NewClip`** (see `PORTING_ADDONS.md`).
2. **Behavior:** Match PB 2022 flow (`PB_Weapon` / `PB_WeaponBase`, tokens, spawner hooks, `KEYCONF` slot defaults) per the **Add a new weapon** recipe above — not bare ZScript ports unless the file you are editing is already ZScript-native.
3. **Spawn toggle:** Every weapon that can appear from a spawner must have an **On/Off** menu toggle (`SpawnOnOff` + `pb_No*` cvar) and all spawn branches must respect it.
4. **HUD:** Any weapon with ammo must have **dedicated** `SBARINFO` `IsSelected` handling for its primary (and secondary, if any) ammo so the fullscreen HUD matches other guns in its family.
5. **Assets:** Copy sounds under `sounds/` (path style used in `SNDINFO.txt` weapon sections) and sprites under `SPRITES/` (preserve lump/sprite names; respect case-sensitive paths on Linux loads).

**Fold equipment (e.g. Molotov cocktail)**  
Follow the same spawn/menu/HUD ideas, but the hook points are: `BaseWeapon.dec` equipment states (`UseEquipment` / `SwitchEquipment` / throw sequence), `zscript/PbWheel/ev_core_special.zsc` `specialWheel_Equipment`, `SBARINFO.txt` `InInventory` (not `IsSelected`), and a `pb_No*` cvar. Shipped assets: OGGs under `sounds/combat/weapons/MolotovCocktail/`, graphics under `SPRITES/weapons/Equipment/MolotovCocktail/`, brightmap PNGs under `Brightmaps/` (MOLO/MOLP) plus `lumps/includes/gldefs/MolotovCocktail.inc` (pulled in by `GLDEFS.txt`), wheel icon `graphics/pywheel/Equip_Molotov.png` (see `SNDINFO.txt` and `zscript/PbWheel/ev_core_special.zsc`).

**Port a newer-PB weapon add-on.** See `PORTING_ADDONS.md` for the compatibility map between 4.11.x-era PB add-ons and this 4.5.0-era codebase, plus the Lever Action port as a worked example.

**Add a CVar / menu option**
1. Declare it in `CVARINFO` (prefer the `pb_`, `cl_`, or `fs_` prefix matching existing conventions).
2. Expose it in `MENUDEF.txt` under the relevant submenu.
3. Add `OPTMNU_*` / menu label strings to `language.enu`.
4. Read it with `CVar.GetCVar("name", plr).GetBool()/GetInt()/GetFloat()` in ZScript, or `GetCVar` in DECORATE/ACS.

**Add a new ZScript event handler**
1. Create the class in `zscript/...` extending `StaticEventHandler` (for cross-level state) or `EventHandler`.
2. `#include` it from `ZSCRIPT.zc`.
3. Add `AddEventHandlers = "ClassName"` inside the `GameInfo { ... }` block in `zmapinfo.txt`.

**Add ACS script**
1. Write/extend in `SRC/<Name>.acs`.
2. Compile with ACC to `ACS/<Name>.o`.
3. Ensure the lump name appears in `LOADACS.txt`.

## 7. Conventions

- **Null checks (`player` / `PlayerInfo`).** On `PlayerPawn`, `self.player` can be **null** when there is no active controller (camera cutaways, possession edges, odd multiplayer transitions). Any use of **`player.cmd`**, **`player.cheats`**, **`player.viewz`**, or **`Player.OnGround`** (same backing pointer) must happen only after **`if (!player) { Super.<override>(); return; }`** at the start of overrides such as **`PlayerThink`**, **`HandleMovement`**, **`CalcHeight`**, and **`CheckPitch`**. Helpers that take **`PlayerInfo`** from **`self.Player`** should **`if (!Player) return <safe default>;`** before reading **`Player.cmd`** (see **`ZMScale()`**). Prefer one guard at the top of the override over scattering checks through movement helpers unless a helper is called from multiple entry points with different assumptions.
- **Weapon actions (`invoker.owner.player`).** `Weapon` / `PB_WeaponBase` action functions that read **`invoker.owner.player`** or **`invoker.owner.player.ReadyWeapon`** must **`if (!invoker.owner || !invoker.owner.player) return …;`** first (safe **`false`** / **`null`** / **`SelectAnimation`** depending on the helper). **`PB_RespectIfNeeded`** and **`PB_MeleeKickInterruptCheck`** / **`PB_WeaponIsInReadyState`** in **`BaseWeapon.zc`** follow this pattern. **`Inventory`** helpers using **`Owner`** (e.g. **`TiltPlusPlus.zc`**) should guard **`Owner`** and **`Owner.player`** (and **`Owner.player.mo`** for view height) before dereferencing.
- **Casing.** Preserve the exact casing used in existing `#include` paths and lump names. Linux GZDoom treats them as case-sensitive.
- **Indentation.** Match the file you're editing: DECORATE files generally use 4-space indents; ZScript files use tabs.
- **Comments.** Do not add comments that narrate what the code does. The existing ZScript comments only explain intent (e.g. `// I call these "Mini Event Handlers" in decorate`).
- **Attribution.** Preserve the header comment `// IF YOU WANT TO USE STUFF FROM THIS MOD, GIVE ME CREDITS` at the top of `DECORATE` and any author credits in sub-files.
- **Reuse existing tokens.** Before defining a new flag-style `Inventory`, search `DECORATE` for an equivalent one.
- **Fatality / execution unblock.** Player states that call `SetPlayerProperty(0,0,0)` to leave a fatality or camera lock should clear **`PROP_TOTALLYFROZEN`** in the same transition (`SetPlayerProperty(0,0,PROP_TOTALLYFROZEN)`), not only property `0` — they are separate engine bits; leftover total freeze causes soft-locks after finishers. See **`CHANGELOG.md`** under **[Unreleased] → Fixed** for the repo-wide sweep (`NEWPLAYE.dec`, `PLAYER.dec`, **`BaseWeapon`**).
- **Third-person execution chase-cam tuning (`CameraRunner.RunCameraNew` / `ReturnCamPos`).** Cinematic framing constants live at the top of `class CameraRunner` in **`ZSCRIPT.zc`**: **`PB_CHASECAM_DISTANCE`** (default `160.0`, map units traced behind the player; effective camera distance is this + `radius*2`) and **`PB_CHASECAM_MAX_DOWN_PITCH`** (default `25.0`, max down-pitch in degrees the chase cam is allowed to inherit from `owner.pitch`). The clamp is applied in **both** `ReturnCamPos` (the `LineAttack` slope) AND `RunCameraNew` (the `SetActorPitch` render pitch) — clamping only one of them produces a mismatched framing where the camera is either too high but level, or correctly placed but tilted into the floor. Upward look (negative pitch) is intentionally unclamped. Wall clipping is handled by `LAF_NOINTERACT`, so the trace still stops at solid geometry — feel free to bump `PB_CHASECAM_DISTANCE` further without worrying about see-through walls in tight corridors.
- **Third-person execution HUD-hide gate (`CameraRunner` inventory).** During third-person executions / fatalities the entire HUD stack — visor + gas-mask, pickup / weapon-mode / dash overlays, **and** the SBARINFO bar — must vanish and snap back the moment the camera reattaches. The signal is the existing **`CameraRunner`** `Inventory` actor (`ZSCRIPT.zc` inline class): every `Fatality*` state in **`actors/Player/NEWPLAYE.dec`** calls `ACS_ExecuteAlways(477,…)` on entry and `ACS_ExecuteAlways(171,…)` on exit, and **`SRC/SCHASECAM.txt`** scripts 477 / 171 do `GiveInventory("CameraRunner",1)` / `TakeInventory("CameraRunner",1)` (script 477 also gates on `r_deathcamera != 0` and `!CheckFlag(0,"CORPSE")` so the death camera doesn't grant it). Three surfaces share this gate and **must stay in lockstep**:
  1. **`zscript/PB_VisorOverlay.zc`** `PB_VisorHandler.RenderUnderlay` — early-return on `players[consoleplayer].mo.CountInv("CameraRunner") > 0` (covers visor corners, glass, center brace, AND the optional gas-mask layer).
  2. **`zscript/PBPDA.zc`** `PB_EventHandler.RenderOverlay` — `bool hideHUD` flag gates `DrawPB2022WeaponPickupOverlay`, `DrawPB2022WeaponModeOverlay`, and the `pb_dasheffect` fullscreen flash. Mouse / PDA-UI bookkeeping below the HUD draws is *intentionally* allowed through so closing the PDA mid-execution doesn't desync.
  3. **`SBARINFO.txt`** — each of the four `statusbar` blocks (`fullscreen`, `normal`, `inventoryfullscreen`, `inventory`) wraps its body in `InInventory CameraRunner { } else { … }` (default amount = 1) so the engine-drawn fullscreen HUD, doom-style status bar, and inventory bars all skip drawing. **Syntax gotcha:** SBARINFO accepts `InInventory <name>` (default amount 1) and `InInventory <name> , <amount>` (comma required — see the existing `InInventory "DMRUpgraded" , 1` reference) but **not** `InInventory <name> <amount>` without the comma — that fails to parse with `Expected '}' but got integer constant '1'`. **Note:** the original `statusbar fullscreen` block had a longstanding stray `}` at end-of-block (extra closing brace beyond the matching `{`); the wrap consumed that stray (the new `else {` now matches it), so brace count is balanced post-edit. If you reshuffle SBARINFO, run a brace-balance sanity check before and after.
  First-person `Execution_*` finishers (`pb_experimental` / `PB_Execute` paths in **`zscript/Weapons/BaseWeapon.zc`** and **`zscript/Weapons/executionGUI/pb_execution_handler.zs`**) are *not* affected because they don't grant `CameraRunner` — only third-person camera-detached finishers hide the HUD.
- **DoomEdNum discipline.** Always update `DoomEdNums` in `zmapinfo.txt` and never reuse an existing number.

## 9. Finding lumps in external GZDoom TCs & optional Brutal / BD22 “folds” (do not clobber core PB)

Use this when auditing **another mod** (e.g. Brutal Pack V10, Brutal Doom 22 test tree) to inventory graphics, models, and sounds, and when **folding** selected pieces into this repo.

### 9.1 How to *search* for content (lessons learned)

- **`glob` / `**/*.*` and similar patterns are a trap.** They usually list only files with a **dot extension**. A huge share of GZDoom content is **extensionless 8.3–style names** (e.g. `D_DM05`, `BSPRG0`, `KPRMA4A6`, root lumps `XHAIRS`, `BOTINFO`) and will **not** appear. Always supplement with a filter for `Extension` empty (or a directory listing that does not require `.`).
- **`.lmp` is a first-class graphic lump type.** Doom-era patches/flats and many TC assets ship as **`*.lmp`** (sometimes beside PNGs). When hunting sprites, brightmaps, or flats, **search for `.lmp` explicitly** — do not assume PNG-only trees.
- **Extensionless graphics still load.** Loose files with **no extension** (raw lumps on disk) are valid for many lumps; GZDoom matches them by lump name. **`SNDINFO`**, **`TEXTURES`**, and **`DECORATE`** reference **logical names**, not necessarily `NAME.png`. When checking whether an asset is present, search by **basename** (ripgrep, recursive directory listing without forcing `*.png`), not only extension-filtered globs.
- **Brightmaps (`doomdefs.bm` / `BMAP`).** PB ships **`map "Brightmaps/ITEMS/<NAME>.png"`** for tier/item brightmaps (see **`BMAP/Armor.txt`** and **`BMAP/WolvusMutatorBonuses.txt`**). UZDoom resolves those paths against the texture manager; missing files log **`Brightmaps/ITEMS/… not found in texture`** at startup (**warning only**, not a VM abort). If upstream only gives extensionless files, **rename or duplicate** them as the referenced **`.png`** beside existing `brightmaps/ITEMS/` art, or register a matching texture — extensionless paths alone often **will not** bind as brightmaps in loose-folder loads unless the engine can resolve that texture name.
- **On Windows, use PowerShell and `-LiteralPath`** for paths with parentheses, spaces, or OneDrive — e.g. `…/temp/(worktrees)/01 bd22test6rc1`. A bare string can break; `-LiteralPath 'C:\…\01 bd22test6rc1'` is safe.
- **Extensionless inventory (examples to run in PowerShell):**
  - `Get-ChildItem -LiteralPath '…' -Recurse -File | Where-Object { -not $_.Extension }` — extensionless only.
  - `Get-ChildItem -LiteralPath '…' -Recurse -File | Where-Object { $_.Extension -match '(?i)\.(md2|md3|png|lmp|wad|pk3|pk7)$' }` — common asset extensions (includes `.lmp`).
- **A folder mod may have no monolithic `*.wad` / `*.pk3` at the root** and still be a full mod: **loose** directories are valid for `-file` loads; don’t treat “no WAD found” as “no assets.”
- **Gore and props are often split** between **`sprites/…`** (loose `.png` in subfolders **or** extensionless image lumps), **`models/…/…md3`**, and **skin/texture `*.png`** next to the MD3. GZDoom ships **`modeldef*.txt`** and matches sprite frames to models via `FrameIndex`. Search for `modeldef` and `MODELS` in the other project.
- **Ripping from embedded WADs** inside a third-party pack: if the tree **does** include `SomePack.wad`, use **SLADE**, **Ultimate Doom Builder** resource browser, or similar to list **embedded** lumps; those never show up in a **recursive file search of only loose folders**.
