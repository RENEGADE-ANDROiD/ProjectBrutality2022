# AGENTS.md — Project Brutality 2022 (main mod)

This file orients coding agents to the Project Brutality add-on that lives in this folder. Read it before editing anything here.

**Recent work (player-facing + maintainer):** see **`CHANGELOG.md`** in this folder. It uses [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) sections (`[Unreleased]`, Added / Changed / Fixed / Removed / Documentation, plus optional pending follow-ups). **After you ship a notable fix or feature**, append a bullet under the right heading in `[Unreleased]` (or under a new dated heading when you tag a release). **`git log`** remains the full history; the changelog is the curated summary for humans and agents.

## 1. What this is

- **Project Brutality 2022** — a large gameplay and content overhaul for *Doom* / *Doom II*, built for **GZDoom** (engine line currently tracked by [UZDoom](https://github.com/UZDoom/UZDoom/tree/4.14.3)).
- Loose folder layout, **not** PK3-packed. GZDoom loads the directory directly (drag the folder onto `gzdoom.exe`, or pass it with `-file`).
- ZScript version pin: `version "4.5.0"` at the top of `ZSCRIPT.zc` — minimum GZDoom ~4.5; development generally targets the latest stable GZDoom.
- Custom player class: **`PB_Doomer`** (locked by `NoRandomPlayerClass = True` in `gameinfo.txt`).
- The **Glory Kills** and **Monster Pack** content was previously shipped as sibling add-on folders and has been folded into this mod. There is nothing external to load on top of the main folder anymore — treat the Glory Kill / pinata system and the extra MP monsters (Crackodemon, Hellduke, Hellsmith, Director, Horrorspawn, Hunger, Aracnorb Queen, Helemental, Hierophant, ESoul, ZombieTank/Plasma/Missile/Elite, PB_ZombieFlyer, etc.) as first-class main-mod content.

## 4. Architecture notes that catch agents off guard

- **Mixed DECORATE + ZScript.** Most actors are still DECORATE (`actors/**/*.dec`). Newer systems — event handlers, HUD UI, wheel UI, gore, PlayerPawn movement, execution GUI — are ZScript (`zscript/**/*.zc|.zsc|.txt`). **Do not port DECORATE to ZScript wholesale.** Follow the convention of the file you're editing.
- **Inventory tokens as flags.** A large block at the top of `DECORATE` defines tiny `Inventory` actors (`EvadeCheck`, `ExecutionToken`, `IsDead`, `PlayerWheelOpen`, `CantFire`, `NoZombieGrenade`, `BrutalCounter`, `FallingHeight`, `TargetIsACyberdemon`, etc.) that are used as booleans/counters through `A_GiveInventory` / `CountInv` / `CheckInventory`. Scan these before adding a new one — many already exist. **Barrel grab / carry** tokens for the Staging-style set (incl. flaming barrels) live in **`actors/Decorations/Barrels/Core/NukageBarrelTokens.dec`** — use **`HasBurningBarrel` / `GrabbedBurningBarrel`** (not `*Flame*`) for compatibility with **`PLAYER.dec`** / **`BaseWeapon`**.
- **Event handlers are registered via MAPINFO**, not in ZScript. See the `GameInfo { AddEventHandlers = ... }` block in `zmapinfo.txt`. Examples: `pb_ExecutionHandler`, `PB_EventHandler`, `MBlurHandler`, `PB_WheelBSSBlurHandler`, `NashGoreHandler`, `BDv22GoreHandler`, `BDv22GoreMergeHandler`, `BPv10GoreHandler`, `PB_DeathGoreBoostHandler`, `TiltPlusPlusHandler`, `DeathFadeBootstrap`, `DEDashJumpHandler`, `GrapplingHookHandler`, `SpeedoMeterHandler`, `WallSlideHandler`, `PB_SpecialWheelHandler`, `HatExtravaganza`, `PB_HitFeedback_Handler`, `HelmetDropHandler`, `DamNumEventHandler`, **`PB_TacticalFeelHandler`**, `KSHUDMessageHandler`, `PB_KillstreakHandler`, `PB_KillstreakRareFlare`, `PB_InstantWeaponSwitchHandler`, `CSSGEvent`, **`PB_FrozenSolidHandler`** — **always prefer `zmapinfo.txt` as source of truth**; the list grows with features.
  Adding a handler requires both a ZScript class **and** a new `AddEventHandlers` line.
- **Grappling hook (`pb_hook`, `GrapplingHookHandler`).** Owning the meat-hook inventory enables the hook; **`pb_hook`** is style only — **`2`** = pendulum-style behaviour on re-fire, otherwise re-fire detaches.
- **Editor numbers** come from the reserved **23000–23133** range in the `DoomEdNums` block of `zmapinfo.txt`. Add a new entry there when introducing a placeable actor; do not reuse numbers. The live lump continues into higher bands (**23134+**, **232xx**) for merged add-ons (Realm667 power-ups, Blood Ghost/Skull GK wrappers, Cat's Frozen content, **`PB_DarkMatterRifle`**, etc.). **`23213` is intentionally unused** (freed slot inside the Cat's Frozen **23212–23218** pickup block — former slice‑5 rocket launcher ednum); **do not reassign** it to unrelated actors.
- **Realm667 monster GK wrappers (`actors/Monsters/Realm667/Realm667_GloryKills.dec`).** `Death.Fatality` RNG is rewritten so Eld tokens honor the sprite silhouette (imps, baby cacos, fat cyber‑manc, flying drone ghost, Blood Ghost). **`PB_Realm667_BlotGK`** / **`PB_Realm667_BloodSkullGK`** still **`GoFatality`** for GK credit/pinata but **`Stop`** without granting FPS finisher inventory—tiny props cannot puppet NATT/NZ0D strips.
- **GLDEFS.txt — commented-out includes / objects.** The lump formerly opened with a **`/* … */`** block listing alternate **`#include "BMAP/*.txt"`** brightmap shards (`Shields`, `Ammo`, `Armor`, `Weapons`, `Carbine`, `AutoShotgun`, `Minigun`, `Nailgun`, `LMG`, `M2Plasma`, `Railgun`, `BHG`, `Revolver`, `Flamecannon`, `Freezer`, `Excavator`, `Throwables`, `VoidSpectre`) — those bulk includes stay **off**; active material/light defs use **`#include "lumps/includes/gldefs/*.inc"`** (plus Nash model materials). Additional **`/* … */`** blocks may wrap disabled **`object`** stubs (e.g. **`PurplePlasmaBall`**) — treat as intentionally off unless you mean to revive them.
- **GLDEFS — dense particle-trail lights.** Long **`PointLight`** chains on some particle trails (historically called out for heavy **`ZWAVE`** / queen-style plasma-trail FX) can dominate frame time — duplicate those rigs sparingly without profiling.
- **DECALDEF — Monster Pack merge collision.** MP decals were folded into the root [`DECALDEF`](DECALDEF). Upstream MP defines **`FireworxScorch`** twice (**`Scorch1`** vs **`Scorch9`** variants); PB keeps a single **`FireworxScorch`** (**`Scorch1`**) and drops the duplicate so the name does not collide at parse time.
- **ACS workflow.** Edit `.acs` in `SRC/`, recompile to `.o` in `ACS/` with **ACC** (matching the GZDoom version), and make sure the lump name is listed in `LOADACS.txt`. Never hand-edit `.o` files. Run **`acc.exe`** (e.g. from SLADE’s bundled ACC) with **`-i`** pointing at the folder that contains **`zcommon.acs`**. Common outputs: `SRC/GloryHUD.acs` → `ACS/GloryHUD.o`, `SRC/GloryChain.acs` → `ACS/GloryMelee.o` (lump name differs from source filename — see `LOADACS.txt`).
- **ACS — guided rocket homing cutoff (`SRC/RLACS.acs`, Scotty).** `setVelocityToTarget` only updates velocity while **`distance > cutoff`** (`ROCKET_CUTOFF`); nearer than that, homing stops so steep downward shots do not oscillate hovering above the floor. **`getTargetPitch`** adapted from ZDoom Wiki **GetTargetPitch**.
- **ACS — DMR zoom reticle aspect (`SRC/Reticle.acs`).** `GetAspectRatio` picks **`169` / `1710` / `1610` / `54` / `43`** from **`GetScreenWidth`/`Height`** tolerances plus **`vid_nowidescreen`** / **`vid_tft`** (typical **`1280×1024`** reads as **`43`** unless **`vid_tft`** marks **`5:4`**); **`1280×800`** forced **`43`**. Feeds scoped **`HudMessage`** drift for **`Rifle` + `Zoomed` + `ReticleStyle==1`** (CATSVISOR drift technique).
- **ACS — low health / armor alerts (`SRC/lowhealth.acs`).** Glory Kills lineage (BoA / GK add-on thresholds). **`lowhealth`** loops **`LOWHEALTH`** while HP ≤ **`sv_LowHealthValue`** when **`be_LowHealthAlert`** on. **`lowarmor`** plays **`LOWARMOR`** once when armor crosses downward through **`sv_LowArmorValue`** after the player had armor above threshold (**`wasEquipped`** latch), so spam is avoided while **`be_LowArmorAlert`** is on.
- **Player class is fixed** to `PB_Doomer`. It is defined in `actors/Player/PLAYER.dec` and extended by `PlayerPawnBase : PlayerPawn` in `zscript/PlayerPawn.zc`. `gameinfo.txt` enforces `NoRandomPlayerClass = True`.
- **Vector `Unit()` on zero length (`zscript/PlayerPawn.zc`).** Normalizing with the engine `Vector2`/`Vector3` `.Unit()` on a zero-length vector yields NaN. `PlayerPawnBase` exposes `SafeUnit2` / `SafeUnit3` that only divides out length when it is non-zero.
- **Painkiller-style air turn cap (`zscript/PlayerPawn.zc`).** `TopSpeedPenality()` (PK movement mode path) reduces `ActualMaxAirSpeed` when large per-tic view yaw deltas arrive and relaxes it slightly on small deltas, then clamps between ground and air caps — an approximation of Painkiller’s “speed vs turn” feel, not a literal port of that engine’s angle range.
- **Weapon next/prev cycling wrap (`zscript/PlayerPawn.zc`).** PB stacks many weapons per slot; vanilla `PickNextWeapon` / `PickPrevWeapon` exit after `NUM_WEAPON_SLOTS` iterations and bail early when slot 0 is empty, which skips wraps and off-list anchors when `LocateWeapon` fails. `PlayerPawnBase` uses `PB_NUM_WP_SLOTS` (10) and `PB_WEAPON_CYCLE_WRAP_CAP` (128) to mirror upstream intent from `player.zs` with a wider search.
- **Helmet visor draw order (`zscript/PB_VisorOverlay.zc`).** `PB_VisorHandler` paints in `RenderUnderlay` so shell/glass/gas-mask sit **under** the ZScript fullscreen bar (`PB2022_Hud_ZS`); using `RenderOverlay` would cover ammo/health/armor. Subclassing `SBarInfoWrapper` was avoided — on non-default wrappers `AttachToPlayer` can read a null internal `script` and crash at startup. Layout targets a virtual 320×200 HUD frame scaled by `Screen.GetHeight()/200` or, when valid, `statusbar.GetHUDScale().y` (same pattern as `SpeedoMeterHandler` in `PlayerPawn.zc`).
- **SBARINFO mugshot + equipment tile (`SBARINFO.txt`).** The fullscreen HUD draws the portrait and equipment frame at **37×37** px to match the **`EQUPCO`** composite sizing in **`Textures.txt`**; legacy **600×900**-style source art blows up unreadably if applied at native resolution.
- **LaserFollower first state tic (`zscript/LaserAPI/LaserAPI.txt`).** Trail sprites need a real state tic (`TNT1 A 1`) before assigning `sprite` from `followSprite`; a `NoDelay`-only spawn path does not reliably pick up the follower graphic.

## 5. Common-change recipes

**Add a new monster (DECORATE path)**
1. Create `actors/Monsters/<Tier>/<Name>.dec` extending `PB_Monster` (or an appropriate sibling).
2. Add `#include "actors/Monsters/<Tier>/<Name>.dec"` near the other monsters in `DECORATE`.
3. Reserve a `DoomEdNum` in `zmapinfo.txt` (next free in 23000–23133 range).
4. Register sounds in `SNDINFO.txt` (append near related monster / weapon blocks).
5. Add sprites under `SPRITES/`, brightmaps in `BMAP/` + bindings in `doommonsters.bm`, and any dynamic lights in `GLDEFS`.
6. If the monster should spawn in vanilla levels, either use `Replaces` on the actor or hook into the appropriate spawner under `actors/SPAWNERS/MonsterSpawners/`.

**Add a new weapon**
1. Create `actors/Weapons/Slot<N>/<Name>.dec` extending `PB_Weapon` (from `actors/Weapons/BaseWeapon.dec`).
2. `#include` it under the matching section in `DECORATE`. If the weapon references a PBWP ammo alias (`PB_Shell`, `PB_HighCalMag`, `PB_Cell`, `PB_CellPack`) by name in state-action args, place the `#include` **after** the `PBWP_CompatAmmo.dec` block near line ~157, not in the trailing weapon block — see the **"DECORATE include order"** bullet in §4.
3. Reserve a `DoomEdNum` in `zmapinfo.txt`.
4. Add ammo (if needed) in `actors/Items/Ammo/` and pickup strings to `language.enu`.
5. Register sounds in `SNDINFO.txt` in the `PBWeapons` (or new weapon) section.
6. Add sprites/graphics and any `TEXTURES.*` patches.
7. If the weapon is ZScript-only (`class …` in `zscript/Weapons/`), make it **`class MyGun : PB_WeaponBase`** and wire **`Select`** (weapon-specific prefix) → **`Goto SelectFirstPersonLegs`** → **`SelectContinue`** (see the **ZScript-only weapons** bullet in §4). ## 7. Conventions

- **Null checks (`player` / `PlayerInfo`).** On `PlayerPawn`, `self.player` can be **null** when there is no active controller (camera cutaways, possession edges, odd multiplayer transitions). Any use of **`player.cmd`**, **`player.cheats`**, **`player.viewz`**, or **`Player.OnGround`** (same backing pointer) must happen only after **`if (!player) { Super.<override>(); return; }`** at the start of overrides such as **`PlayerThink`**, **`HandleMovement`**, **`CalcHeight`**, and **`CheckPitch`**. Helpers that take **`PlayerInfo`** from **`self.Player`** should **`if (!Player) return <safe default>;`** before reading **`Player.cmd`** (see **`ZMScale()`**). Prefer one guard at the top of the override over scattering checks through movement helpers unless a helper is called from multiple entry points with different assumptions.
- **Weapon actions (`invoker.owner.player`).** `Weapon` / `PB_WeaponBase` action functions that read **`invoker.owner.player`** or **`invoker.owner.player.ReadyWeapon`** must **`if (!invoker.owner || !invoker.owner.player) return …;`** first (safe **`false`** / **`null`** / **`SelectAnimation`** depending on the helper). **`PB_RespectIfNeeded`** and **`PB_MeleeKickInterruptCheck`** / **`PB_WeaponIsInReadyState`** in **`BaseWeapon.zc`** follow this pattern. **`Inventory`** helpers using **`Owner`** (e.g. **`TiltPlusPlus.zc`**) should guard **`Owner`** and **`Owner.player`** (and **`Owner.player.mo`** for view height) before dereferencing.
- **Casing.** Preserve the exact casing used in existing `#include` paths and lump names. Linux GZDoom treats them as case-sensitive.
- **Indentation.** Match the file you're editing: DECORATE files generally use 4-space indents; ZScript files use tabs.
- **Comments.** Do not add comments that narrate what the code does. The existing ZScript comments only explain intent (e.g. `// I call these "Mini Event Handlers" in decorate`).
- **Attribution.** Keep licensing / author credits in **`CREDITS.txt`** and per-folder **`ASSET_NOTES.txt`** where they exist; do not strip those files during comment audits. The root **`DECORATE`** lump no longer carries a standalone credits banner — rely on **`CREDITS.txt`** instead.
- **Reuse existing tokens.** Before defining a new flag-style `Inventory`, search `DECORATE` for an equivalent one.
- **Fatality / execution unblock.** Player states that call `SetPlayerProperty(0,0,0)` to leave a fatality or camera lock should clear **`PROP_TOTALLYFROZEN`** in the same transition (`SetPlayerProperty(0,0,PROP_TOTALLYFROZEN)`), not only property `0` — they are separate engine bits; leftover total freeze causes soft-locks after finishers. See **`CHANGELOG.md`** under **[Unreleased] → Fixed** for the repo-wide sweep (`NEWPLAYE.dec`, `PLAYER.dec`, **`BaseWeapon`**).
- **Third-person execution chase-cam tuning (`CameraRunner.RunCameraNew` / `ReturnCamPos`).** Cinematic framing constants live at the top of `class CameraRunner` in **`ZSCRIPT.zc`**: **`PB_CHASECAM_DISTANCE`** (default `160.0`, map units traced behind the player; effective camera distance is this + `radius*2`) and **`PB_CHASECAM_MAX_DOWN_PITCH`** (default `25.0`, max down-pitch in degrees the chase cam is allowed to inherit from `owner.pitch`). The clamp is applied in **both** `ReturnCamPos` (the `LineAttack` slope) AND `RunCameraNew` (the `SetActorPitch` render pitch) — clamping only one of them produces a mismatched framing where the camera is either too high but level, or correctly placed but tilted into the floor. Upward look (negative pitch) is intentionally unclamped. Wall clipping is handled by `LAF_NOINTERACT`, so the trace still stops at solid geometry — feel free to bump `PB_CHASECAM_DISTANCE` further without worrying about see-through walls in tight corridors. **`CameraRunner`** is granted/revoked by ACS scripts **477** / **171** in **`SRC/SCHASECAM.txt`** on **`Fatality*`** entry/exit from **`actors/Player/NEWPLAYE.dec`**. The **`SpawnerBase`** / **`Demonpickup.SpawnActor`** helpers in the same lump mirror an upstream spawn-flag forwarding pattern historically credited **Marisa Kirisame**; chase-cam logic itself descends from an ACS prefab ported by **Cacodemon345**.
- **Third-person execution HUD-hide gate (`CameraRunner` inventory).** During third-person executions / fatalities the entire HUD stack — visor + gas-mask, pickup / weapon-mode / dash overlays, **and** the SBARINFO bar — must vanish and snap back the moment the camera reattaches. The signal is the existing **`CameraRunner`** `Inventory` actor (`ZSCRIPT.zc` inline class): every `Fatality*` state in **`actors/Player/NEWPLAYE.dec`** calls `ACS_ExecuteAlways(477,…)` on entry and `ACS_ExecuteAlways(171,…)` on exit, and **`SRC/SCHASECAM.txt`** scripts 477 / 171 do `GiveInventory("CameraRunner",1)` / `TakeInventory("CameraRunner",1)` (script 477 also gates on `r_deathcamera != 0` and `!CheckFlag(0,"CORPSE")` so the death camera doesn't grant it). Three surfaces share this gate and **must stay in lockstep**:
  1. **`zscript/PB_VisorOverlay.zc`** `PB_VisorHandler.RenderUnderlay` — early-return on `players[consoleplayer].mo.CountInv("CameraRunner") > 0` (covers visor corners, glass, center brace, AND the optional gas-mask layer).
  2. **`zscript/PBPDA.zc`** `PB_EventHandler.RenderOverlay` — `bool hideHUD` flag gates `DrawPB2022WeaponPickupOverlay`, `DrawPB2022WeaponModeOverlay`, and the `pb_dasheffect` fullscreen flash. Mouse / PDA-UI bookkeeping below the HUD draws is *intentionally* allowed through so closing the PDA mid-execution doesn't desync.
  3. **`SBARINFO.txt`** — each of the four `statusbar` blocks (`fullscreen`, `normal`, `inventoryfullscreen`, `inventory`) wraps its body in `InInventory CameraRunner { } else { … }` (default amount = 1) so the engine-drawn fullscreen HUD, doom-style status bar, and inventory bars all skip drawing. **Syntax gotcha:** SBARINFO accepts `InInventory <name>` (default amount 1) and `InInventory <name> , <amount>` (comma required — see the existing `InInventory "DMRUpgraded" , 1` reference) but **not** `InInventory <name> <amount>` without the comma — that fails to parse with `Expected '}' but got integer constant '1'`. **Note:** the original `statusbar fullscreen` block had a longstanding stray `}` at end-of-block (extra closing brace beyond the matching `{`); the wrap consumed that stray (the new `else {` now matches it), so brace count is balanced post-edit. If you reshuffle SBARINFO, run a brace-balance sanity check before and after.
  First-person `Execution_*` finishers (`pb_experimental` / `PB_Execute` paths in **`zscript/Weapons/BaseWeapon.zc`** and **`zscript/Weapons/executionGUI/pb_execution_handler.zs`**) are *not* affected because they don't grant `CameraRunner` — only third-person camera-detached finishers hide the HUD.
- **DoomEdNum discipline.** Always update `DoomEdNums` in `zmapinfo.txt` and never reuse an existing number.

## 9. Finding lumps in external GZDoom TCs & optional Brutal / BD22 “folds” (do not clobber core PB)

Use this when auditing **another mod** (e.g. Brutal Pack V10, Brutal Doom 22 test tree) to inventory graphics, models, and sounds, and when **folding** selected pieces into this repo.

### 9.1 How to *search* for content (lessons learned)

- **`glob` / `**/*.*` and similar patterns are a trap.** They usually list only files with a **dot extension**. A huge share of GZDoom content is **extensionless 8.3–style names** (e.g. `D_DM05`, `BSPRG0`, `KPRMA4A6`, root lumps `XHAIRS`, `BOTINFO`) and will **not** appear. Always supplement with a filter for `Extension` empty (or a directory listing that does not require `.`).
- **`.lmp` is a first-class graphic lump type.** Doom-era patches/flats and many TC assets ship as **`*.lmp`** (sometimes beside PNGs). When hunting sprites, brightmaps, or flats, **search for `.lmp` explicitly** — do not assume PNG-only trees.
- **Extensionless graphics still load.** Loose files with **no extension** (raw lumps on disk) are valid for many lumps; GZDoom matches them by lump name. **`SNDINFO`**, **`TEXTURES`**, and **`DECORATE`** reference **logical names**, not necessarily `NAME.png`. When checking whether an asset is present, search by **basename** (ripgrep, recursive directory listing without forcing `*.png`), not only extension-filtered globs.
- **Brightmaps (`doomdefs.bm` / `BMAP`).** PB ships **`map "Brightmaps/ITEMS/<NAME>.png"`** for tier/item brightmaps (see **`BMAP/Armor.txt`** — includes **DarkShadow** **Custom Red Armor** **`4RM3*`** alongside standard armor pickups — and **`BMAP/WolvusMutatorBonuses.txt`** for elite Wolvus armor). UZDoom resolves those paths against the texture manager; missing files log **`Brightmaps/ITEMS/… not found in texture`** at startup (**warning only**, not a VM abort). If upstream only gives extensionless files, **rename or duplicate** them as the referenced **`.png`** beside existing `brightmaps/ITEMS/` art, or register a matching texture — extensionless paths alone often **will not** bind as brightmaps in loose-folder loads unless the engine can resolve that texture name.
- **On Windows, use PowerShell and `-LiteralPath`** for paths with parentheses, spaces, or OneDrive — e.g. `…/temp/(worktrees)/01 bd22test6rc1`. A bare string can break; `-LiteralPath 'C:\…\01 bd22test6rc1'` is safe.
- **Extensionless inventory (examples to run in PowerShell):**
  - `Get-ChildItem -LiteralPath '…' -Recurse -File | Where-Object { -not $_.Extension }` — extensionless only.
  - `Get-ChildItem -LiteralPath '…' -Recurse -File | Where-Object { $_.Extension -match '(?i)\.(md2|md3|png|lmp|wad|pk3|pk7)$' }` — common asset extensions (includes `.lmp`).
- **A folder mod may have no monolithic `*.wad` / `*.pk3` at the root** and still be a full mod: **loose** directories are valid for `-file` loads; don’t treat “no WAD found” as “no assets.”
- **Gore and props are often split** between **`sprites/…`** (loose `.png` in subfolders **or** extensionless image lumps), **`models/…/…md3`**, and **skin/texture `*.png`** next to the MD3. GZDoom ships **`modeldef*.txt`** and matches sprite frames to models via `FrameIndex`. Search for `modeldef` and `MODELS` in the other project.
- **Ripping from embedded WADs** inside a third-party pack: if the tree **does** include `SomePack.wad`, use **SLADE**, **Ultimate Doom Builder** resource browser, or similar to list **embedded** lumps; those never show up in a **recursive file search of only loose folders**.

## 10. PDA dossier preview strips & `PBPDA.zc` quirks

Authoritative reference for **`zscript/PBPDA.zc`** strip builders and UI scope. **Blocky tile grid, cursor scaling, toggle debounce, and most menu behaviour** are already spelled out in §4 (*PDA toggle + Blocky tile-grid layout*); prefer §4 for Gearbox-parity layout prose — this section covers **preview timelines**, **ZScript pin rules**, and **save/input edge cases** so phase-3 comment sweeps can drop duplicated `//` essays in-source.

### 10.1 Strips and playback

Preview builds parallel **`TextureID`** + per-frame tic arrays (**`pdaPrevFrames`**, **`pdaPrevFrameTics`**, **`pdaPrevSegIdx`** when used) in play scope; **`PB_EventHandler.RenderOverlay`** draws the current frame; **`WorldTick`** steps **`PDA_Play_AdvancePreviewAnim`**. Equipment composes **`PDA_EquipStrip_*`**; monsters use **`PDA_Play_BuildMonster`** / **`PDA_Play_AppendChain`** with **`isMonster=true`** to skip **`PLAY*`** sprites that leak from Death → Glory Kill / fatality choreography.

### 10.2 `PB_PDA_DESC_BREAK_WIDTH`

**`SmallFont.BreakLines`** width is font-native units. Live dossier scale comes from **`EnsurePdaDescAutosizeScale`** (baseline **`pdaTextScale * 0.68 * 0.88`**, clamped to the pane). Blocky mode narrows the dossier column for the right-hand preview (**`PB_PDA_BlockyDescClipRight()`** vs **`descriptiondialog.ImagePos.x`**). **`PB_PDA_DESC_BREAK_WIDTH`** (currently **180**) replaced older **240 → 160 → 104** trials; **104** left ~50% unused horizontal slack in the pane.

### 10.3 Path A / Path B (`PDA_Preview_*`)

**`PDA_WeaponStrip_AppendFullWeapon`** walks **`Spawn`**, **`AppendReady`**, then **`PDA_WeaponStrip_AppendPrimaryFireWithDispatchFallbacks`** along **`NextState`** until **`PDA_Main_FireWalk_ShouldStopBeforeState`** blocks entry into shared **`PB_Weapon`** overlays (**`Flash*`**, kicks, **`Execution_*`**, etc.). If **`Fire`** is dispatch-only or the graph traps the walker, author linear **`PDA_Preview_*`** chains on the weapon (real frames, **`Stop`** only, **no `goto`** into gameplay). Register with **`PDA_WeaponStrip_Append<Class>Preview`** + **`nm == 'ClassName'`** early return in **`AppendFullWeapon`**. **`HellPistoler`**, **`PB_CryoShotgun`**, **`PB_BFG9000`** (`PDA_WeaponStrip_AppendPB_BFG9000Preview`), etc. are the reference implementations.

### 10.4 Fire-walk stop vs `InStateSequence`

**`PDA_FireWalk_NextIsLabelEntry`** avoids false stops: **`InStateSequence(nx, label)`** is true for **`Goto`** tails (e.g. **`FlashAirKicking` → `Ready3`**), which would truncate some weapons before PSP art appears. Stop when **`nx`** is the named label’s **first** state for generic overlay labels; keep **`InStateSequence`**-based stops for **`Execution_*`** roots.

### 10.5 ZScript `version "4.5.0"` pin (`PBPDA.zc`)

- Dynamic state labels: **`def.FindStateByString(label, true)`** — **`FindState('Literal')`** is compile-time only.
- Class-from-string: cast **`Name`** → **`(class<Actor>)(nm)`**, not raw **`String`**.
- **`readonly<>`** templates: no **`readonly<class<Actor>>`** — use **`GetDefaultByType((class<Actor>)(nm))`** into **`readonly<Actor>`** (same for **`Weapon`** / **`Inventory`**).
- **`Array<Class<Actor> >`**: interior space before **`>`** (parser rejects **`>>`**).
- **`PDA_Main_PushTo`**: **`int target`** + **`switch`** writes class fields — bare **`Array<T>`** out-parameters aborted the VM at call binding on this pin.
- **`TextureID` “missing” sentinel**: default **`TextureID`** is invalid; **`null`** is not assignable — helpers return a sentinel texnum for “no frame”.
- **`TexMan.GetScaledSize`**: assign into **`Vector2`** before **`.X`/`.Y`** (no chained access on the return value).

### 10.6 Blocky caches & mouse hitboxes

Positive / negative Blocky icon caches use parallel **`Array<String>` + `Array<TextureID>`** (and **`blocky*Missed`**) — no **`Map<>`**. **`InitializeUiObject`** hitbox sizes must use the same **`pdaTextScale`** anchor as **`RenderOverlay` `DTA_ScaleX/Y`** or **`UiTick`** misses clicks. **`ResolveBlockyWeaponIconTex`** checks **`Inventory.Icon`** before walking **`Spawn`** so an explicit icon survives **`PDA_ShouldSuppressWeaponPlayerSideArt`** on pickup strips (e.g. **`HellPistoler`** **`ARMZ`** on the denylist for animation walks but valid as the static tile).

### 10.7 Equipment unlock poll / debug / save-load

- Equipment codex: **`WorldTick`** walks **`owner.Inv`** every **7** tics (**`GiveInventory`** is not virtual on 4.5.0); linked list field is **`Inv`**, not **`Next`**. **`NotifyEquipmentInventoryGranted`** is idempotent via **`PDAEquipFound`**.
- **`PDA_DebugOn()`** is hard-wired **`false`** so old **`pb_pda_debug=1`** configs don’t spam **`[PDA-DBG]`**; re-enable by rewiring the function to read the cvar.
- Save load: PDA **`transient ui`** caches reset; **`uiObjects.Size() < 1`** triggers **`InitializeUiObjects()`** — removed **`ClearArrays`** thrash (play/ui scope mismatch caused per-frame rebuild).

### 10.8 Input / HUD gate

- **`InputProcess`** emits **`"input"`** netevents; **`NetworkProcess`** integrates **`MouseX`/`MouseY`** — required for PDA cursor motion while frozen.
- **`CameraRunner`** gates HUD overlays during third-person executions (ACS 477/171); **`RenderOverlay`** allows PDA bookkeeping below the gate so closing mid-execution doesn’t desync — same contract as **`PB_VisorHandler`** + **`SBARINFO`** (§7).

### 10.9 Title startup / programmable mice

**`InterfaceProcess` / mouse init:** defer aggressive PDA mouse work immediately after cold start — historic GZDoom + programmable-mouse corruption on Windows (see Project Brutality issue **#341** threads on GitHub). PDA stays disabled on the title screen.

**`InputProcess`:** returns **`false`** when **`gamestate == GS_TITLELEVEL`** or **`gametic < PB_PDA_UI_INIT_MIN_GAMETIC`** (35) so UI-scope mouse handling never runs on the first ticks after launch (same **#341** cold-start family). **`InitializeUiObjects()`** is **not** called from **`RenderOverlay`** until **`PDAMode`** (or a forced codex-unlock path) — avoids a one-shot UI-thread stall / busy cursor on cold start. When **`PDAMode`** is false, events fall through for normal view-turn / firing / menus; when true, mouse motion / **`Mouse1`** down-up emit **`"input"`** netevents and the handler returns **`true`** to consume them so they are not fed into player aim or weapon fire.
